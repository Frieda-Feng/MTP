| 2026-07-05 | 3D 状态立方体的观察角度不理想，部分顶点重叠 | 调整 `ax.view_init(elev, azim)` 参数，多次尝试后选择 elev=25°, azim=-60°，使 8 个顶点和棱边清晰可辨 | 3D 可视化的视角选择需要实际渲染后反复调整，不能仅凭代码判断 |
| 2026-07-05 | 单元测试用 `atol=1e-6` 断言"迭代解 ≈ 直接解"失败——Jacobi 与直接解最大偏差达 1.9×10⁻⁴，比容差大近 200 倍，一度怀疑迭代实现有 bug | 重新审视停止条件：`tol=1e-6` 是"步差"（相邻两轮解之差）而非"与真解的偏差"。查阅数值分析资料，推出线性收敛迭代法的真误差上界 $\lVert x_k-x^*\rVert \le \text{tol}/(1-\rho)$；代入 30×30 铁板 Jacobi 谱半径 $\rho=0.9949$，得理论上界 1.95×10⁻⁴，与实测 1.94×10⁻⁴ 几乎吻合 | 修改测试容差为 `atol=5e-4`（留约 2.5× 余量）后通过。这不是代码 bug，是迭代法的固有数学性质：$\rho$ 接近 1 时放大因子 $1/(1-\rho)$ 很大。教训：**"步差"与"真误差"是两个概念**，混淆会误判迭代精度；若要真误差达 $10^{-6}$，容差需收紧到约 $5\times10^{-9}$。这条结论已写入报告第 4.3 节，成为结果分析的关键论据 |
| 2026-07-04 | $B=10000$ 次重采样用 Python `for` 循环逐次抽样 + 算中位数，单次分析要跑几十秒，覆盖率实验（$R\times B$）更跑不动 | 改用 numpy 向量化：`rng.integers` 一次生成 $(B,n)$ 整数索引矩阵，`np.median(axis=1)` 一次性算出 $B$ 个 bootstrap 中位数；中位数内部用 `np.partition` 保持 $O(n)$ | $B=10000$ 秒级跑完。教训：numpy 下百万级运算必须向量化，不能写 Python 层循环。 |
| 2026-07-05 | BCa 区间和 percentile **几乎重合**（BCa [22.0, 75.5] 与 percentile [22.0, 79.0]），第一反应是代码写错了 | 推导后发现不是 bug：偶数 $n$ 下留一法中位数只取 2 个值 $\{x_{(n/2)},x_{(n/2+1)}\}$，导致加速度分子 $\sum(\cdot)^3 = 0$（偶数 $n$ 无结时精确为 0）、$\hat a\equiv 0$；对 $n\in\{12,24,48,96\}$ 扫描验证都一样 | 确认是中位数（非光滑泛函）的固有性质，BCa 退化为「percentile + 偏差修正」。反思：先怀疑代码、再怀疑方法，本题恰是方法本身的性质，报告须如实承认局限。 |
| 2026-07-05 | Exp(1) 覆盖率 **0.934 < 名义 0.95**，一度怀疑方法不对 | 查文献确认 percentile 一阶精度 $O(n^{-1/2})$ 在偏态数据下本就轻度欠覆盖；MC 误差按观测 $\hat p$ 算约 0.79 个百分点；用解析标尺 SE≈$1/\sqrt{24}$≈0.204、区间宽≈0.80 与实测 0.796 交叉验证吻合 | 是发现不是 bug。教训：模拟结果要有解析标尺交叉验证，不能只盯一个数。 |
| 2026-07-06 | 在 `mission3/` 文件夹里跑 `python -m mission3.run_all` 报 `No module named 'mission3'`，数据相对路径也跟着错 | 这是 Python 包机制限制——模块错误本身无法在代码里消解，改为按运行位置切换命令：仓库根用 `python -m mission3.run_all`、`mission3/` 内用 `python -m run_all`；同时在 `run_all` 入口做 cwd 无关处理（自动定位仓库根与 `data/` 目录、数据加载改用基于包位置的安全路径），让数据路径不再依赖当前目录 | 从仓库根或 `mission3/` 内都能一键跑（分别用对应命令）。教训：可复现性不止固定随机种子，「在哪运行都能跑」也是基本工程要求。 |

# 七、总结与反思

通过本次任务二的实践，主要收获如下：

（1）**数学建模能力**：将一个实际的工程需求（三控开关）转化为离散数学问题，通过枚举状态空间、建立真值表、推导布尔表达式完成逻辑建模。体会到"异或"运算不是凭空选择的——它是由"任一开关变化 → 输出变化"这一设计约束自然导出的唯一解。

（2）**编程实现能力**：使用 Python 从零实现了真值表自动生成、交互模式、自动验证和可视化四个模块。过程中遇到了 Windows 控制台编码不兼容 emoji 的问题，通过替换为 ASCII 符号解决；也遇到了 matplotlib 中文不显示的经典问题，通过配置系统字体和清除缓存解决。

（3）**可视化表达能力**：学会了用 matplotlib 绘制三维散点图、热力图等，理解了三类图（结构 / 过程 / 结果）各自的定位——真值表揭示静态映射，状态空间图揭示拓扑结构，切换路径图揭示动态行为，验证矩阵给出量化结论。四张图相互配合，形成完整的可视化论证链条。

（4）**交互式展示能力**：意识到静态图表在课堂演示和答辩中的局限性，主动学习了 HTML+CSS+JS 前端技术，编写了自包含的交互式演示页面。该页面实现了开关点击切换、灯发光动画、公式逐步展开和真值表实时高亮，让数学模型从"纸上公式"变成"可触摸的实物"，显著提升了展示效果和互动体验。

- 三控开关逻辑可以进一步用硬件描述语言（如 Verilog）实现，对比软件实现更有工程深度。
